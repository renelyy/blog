# CAP、BASE 与一致性

[← 返回索引](./index)

> **本节目标：** 理解 CAP、BASE 与一致性级别，为幂等、锁、事务选型打基础。

---

## ⭐ 从单机到分布式

单机事务靠 **ACID**；多节点、网络分区后，**无法同时** 满足所有理想属性 — 需要 **权衡**。

```text
用户 → 网关 → 服务 A → MySQL
              ↘ 服务 B → Redis
              ↘ 服务 C → MQ → 服务 D
```

**新问题：** 网络延迟、节点宕机、消息乱序、重复投递、时钟漂移。

---

## ⭐ CAP 定理

在 **发生网络分区（P）** 时，系统只能在 **C** 与 **A** 之间二选一：

| 字母 | 含义 |
|------|------|
| **C** Consistency | 所有节点同一时刻看到相同数据（线性一致） |
| **A** Availability | 每个请求都能得到响应（非错即答） |
| **P** Partition tolerance | 网络分区时系统仍能继续工作 |

**结论：** 分布式系统 **必须接受 P**，实际在 **CP** 与 **AP** 间选择。

| 类型 | 例子 | 场景 |
|------|------|------|
| **CP** | ZooKeeper、etcd、HBase | 选主、配置、强一致元数据 |
| **AP** | Cassandra、Dynamo 风格、最终一致缓存 | 高可用读写、可容忍短暂不一致 |
| **CA** | 传统单机 DB（无分区） | 非分布式 |

**误区：** CAP 不是「三选二永远固定」，而是 **分区发生时** 的取舍；正常运行时可同时做好 C 和 A。

---

## ⭐ 一致性级别

| 级别 | 说明 | 典型 |
|------|------|------|
| **强一致** | 写后读立即可见 | 单库事务、ZK |
| **线性一致** | 全局有序，像单机 | etcd、Spanner |
| **顺序一致** | 同一进程内有序 | Kafka 分区内 |
| **因果一致** | 有因果关系的操作有序 | 部分分布式 DB |
| **最终一致** | 一段时间后收敛到相同 | 缓存 + MQ、主从复制 |

**企业默认：** 核心业务库 **强一致（单库事务）**；跨服务、缓存、搜索索引 **最终一致**。

---

## ⭐ BASE 理论

CAP 的 AP 方向延伸：

| 字母 | 含义 |
|------|------|
| **BA** Basically Available | 基本可用 — 降级、限流后仍部分可用 |
| **S** Soft state | 软状态 — 中间态允许短暂不一致 |
| **E** Eventually consistent | 最终一致 — 异步补偿、对账 |

```text
下单（订单服务）→ 发 MQ → 库存服务扣减 → 积分服务加分
         │                              │
         └──────── 短暂不一致 ──────────┘
                    ↓
              对账 / 补偿 / 重试
```

与 [消息可靠性](../messaging/04-reliability-patterns) 衔接。

## ⭐ 复制与一致性（MySQL 视角）

| 模式 | 说明 |
|------|------|
| **主从异步复制** | 主库写完即返回，从库延迟 — **最终一致** |
| **半同步** | 至少一个从库 ACK — 降低丢数据风险 |
| **读写分离** | 读从库可能 **读到旧数据** — 写后读走主库或延迟读 |

```java
// 写后立刻读自己的数据 — 走主库
@Transactional
public void updateProfile(UserProfile profile) {
  userRepo.save(profile);
}

@Transactional(readOnly = true)
public UserProfile getProfile(Long userId) {
  // 刚更新后读 — 若走从库可能旧数据；可强制主库或 ThreadLocal 路由
  return userRepo.findById(userId).orElseThrow();
}
```

见 [InnoDB 与 MVCC](../database/04-innodb-and-mvcc)、[读写分离](../database/06-sharding-readwrite)。

## ⭐ PACELC（延伸）

CAP 的补充：**无分区（E）时**，在 **延迟（L）** 与 **一致（C）** 间权衡。

- **PA/EL** — 可用优先、低延迟（Dynamo、Cassandra）
- **PC/EC** — 一致优先（传统 RDBMS、ZK）

面试与架构讨论常用，不必死记，理解 **「没有免费午餐」** 即可。

---

## 📌 企业如何落地

| 场景 | 策略 |
|------|------|
| 账户余额、库存扣减 | 单库事务或 **悲观/乐观锁** + 幂等 |
| 跨服务扣库存 | **本地消息表 / MQ** + 最终一致 |
| 缓存 | Cache-Aside + TTL + 延迟双删 |
| 配置、选主 | CP 组件（Nacos/ZK） |
| 搜索索引 | 异步同步 ES，允许秒级延迟 |

**原则：** 能单库就不分布式事务；能最终一致就不 2PC。

## ⚠️ 常见坑

| 坑 | 说明 |
|----|------|
| 所有数据都强一致 | 成本极高，多数场景不需要 |
| 读从库当强一致 | 主从延迟导致脏读 |
| 忽略时钟 | 分布式 ID、过期应用 NTP |
| CAP 教条 | 实际按业务 SLA 设计 |
| 缓存当数据源 | 必须 DB 为准 + 过期策略 |

---

## 本章小结

- 分区下 **CP vs AP**；业务上 **强一致 vs 最终一致** 分区使用
- BASE：基本可用 + 软状态 + 最终一致
- 主从、缓存、MQ 都是一致性设计的一部分
